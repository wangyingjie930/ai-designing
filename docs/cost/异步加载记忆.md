这句话要拆开理解，而且我之前说得太笼统了：

> “没完成就跳过、不阻塞”只在当前源码里明确适用于记忆检索；技能发现、工具摘要并不完全一样。

## 先理解“后台运行”

这里的“后台”不是新线程，而是先启动一个异步 `Promise`，暂时不 `await`：

```ts
const promise = 开始检索记忆()

// 不等待检索，马上继续
await 主模型流式输出()
await 执行工具()
```

相当于两条路线同时推进：

```text
主路线：模型流式输出 ──→ 工具执行 ──→ 下一轮模型
辅助线：      记忆检索 ─────────────────→
```

这样主模型输出和工具执行花费的几秒钟，就可以用来“藏住”记忆检索延迟。

## 记忆检索是怎么“没完成就跳过”的

每个用户轮次开始时，Claude Code 启动一次记忆检索：

```ts
using pendingMemoryPrefetch =
  startRelevantMemoryPrefetch(messages, toolUseContext)
```

见 [query.ts](/Users/wangyingjie/Documents/code/claude-code-source-study/src/query.ts:297)。

内部保存两个东西：

```ts
{
  promise,          // 真正的记忆检索任务
  settledAt: null   // null 表示还没完成
}
```

检索完成后，只修改状态：

```ts
void promise.finally(() => {
  handle.settledAt = Date.now()
})
```

见 [attachments.ts](/Users/wangyingjie/Documents/code/claude-code-source-study/src/utils/attachments.ts:2406)。

工具执行结束后，主流程检查：

```ts
if (pendingMemoryPrefetch.settledAt !== null) {
  const memories = await pendingMemoryPrefetch.promise
  // 把记忆注入下一轮模型请求
}
```

注意这里先检查 `settledAt`：

```text
已经完成
  → await 一个已经完成的 Promise
  → 几乎不产生等待
  → 注入记忆

尚未完成
  → 不进入 if
  → 根本不 await
  → 直接继续下一轮
```

对应源码在 [query.ts](/Users/wangyingjie/Documents/code/claude-code-source-study/src/query.ts:1592)。

## 用时间线理解

假设模型和工具一共执行了 3 秒。

### 情况一：检索只花了 800ms

```text
0ms       启动记忆检索
0-2000ms  模型流式输出
800ms     记忆检索已经完成
2000-3000ms 执行工具
3000ms    检查结果：已完成，立即注入
```

用户没有额外等待记忆检索的 800ms，因为它被模型输出时间覆盖了。

### 情况二：检索花了 5 秒

```text
0ms       启动记忆检索
0-2000ms  模型流式输出
2000-3000ms 执行工具
3000ms    检查结果：尚未完成
           ↓
         不等待，继续下一轮模型
5000ms    记忆检索完成
```

如果后面还有一轮工具循环，下一次检查时可以使用这份结果；如果整个任务已经结束，则清理时会取消未完成任务：

```ts
[Symbol.dispose]() {
  controller.abort()
}
```

所以这里的“跳过”是：

> 本次检查点暂时不用，不代表立即取消；后面还有循环就再检查，整个任务结束才取消。

## 三类辅助工作的真实区别

| 辅助工作 | 什么时候启动 | 收集时未完成怎么办 |
|---|---|---|
| 记忆检索 | 用户轮次开始 | 明确零等待跳过，下次循环再检查 |
| 技能发现 | 每次循环、模型请求前 | 当前调用点使用了 `await`；不能笼统说一定跳过 |
| 工具摘要 | 一批工具执行完成后 | 与下一轮模型流式输出重叠；下一轮流结束后仍会 `await` |

工具摘要的流程是：

```text
第一轮工具执行完成
    ↓
启动 Haiku 生成“Read config.json”之类的摘要
    ↓
不等待，马上开始下一轮模型请求
    ↓
下一轮模型流式输出完成
    ↓
await 工具摘要
```

源码明确写的是：

```ts
const summary = await pendingToolUseSummary
```

见 [query.ts](/Users/wangyingjie/Documents/code/claude-code-source-study/src/query.ts:1054)。

因此准确说法应该是：

> Claude Code 会把辅助模型调用与主模型、工具执行重叠，以隐藏大部分延迟；其中记忆预取采用严格的零等待收集，工具摘要则采用“先并发、稍后等待”，技能发现从当前可见调用点看也存在 `await`。

一句话记忆：

> 先偷偷开工，等主流程走到检查点再看；记忆没做完就先不用，工具摘要没做完则可能稍等一下。
